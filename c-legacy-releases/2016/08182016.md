# Notes de mise à jour - Août 2016

Découvrez les nouvelles fonctionnalités et les correctifs d’Adobe Marketing Cloud.

>[!NOTE]
>
>pour recevoir les notes de mise à jour une semaine avant la mise à jour mensuelle, abonnez-vous à [Adobe Priority Product Update](https://www.adobe.com/subscription/priority-product-update.html), où les informations de mise à jour sont disponibles une semaine avant la date de publication. Revenez-y au moment de la publication de la mise à jour pour connaître les dernières informations.

Dernière mise à jour : 25 août 2016

## Activation 

### Service Marketing Cloud ID {#mcvid}

Cette version est numérotée 1.7.0. Elle comprend les fonctionnalités, correctifs de bogues et autres modifications ci-après.

**Correctifs et améliorations**

* Ajout de ` idSyncAttachIframeOnWindowLoad` comme indicateur booléen en option que vous pouvez définir dans la fonction ` Visitor.getInstance`. Si ` idSyncAttachIframeOnWindowLoad= true`, le service d’identification charge l’iFrame de synchronisation des identifiants au chargement de la fenêtre. Par défaut, le service d’identification charge l’iFrame aussi vite que possible. Cet indicateur *remplace*` idSyncAttachIframeASAP`, qui est obsolète. Voir [Variables utilisées par la fonction getInstance du service d’ID](https://marketing.adobe.com/resources/help/en_US/mcvid/mcvid-variables.html). (MCID-122)
* Ajout d’une fonctionnalité permettant la prise en charge du suivi des [!DNL  Marketing Cloud] ID à l’échelle de tous les domaines, applications natives et applications hybrides jusqu’aux transitions web. Voir [Fonction auxiliaire d’ajout d’identifiant visiteur](https://marketing.adobe.com/resources/help/en_US/mcvid/mcvid-appendvisitorid.html). (MCID-42)
* Ajout de fonctions au code visitorAPI.js qui déterminent si le service d’ID a généré l’[!DNL  Marketing Cloud] ID du visiteur côté client ou côté serveur ou si les appels d’identifiants sont expirés. Voir les sections relatives aux [fonctions de suivi de dépassement de délai](https://marketing.adobe.com/resources/help/en_US/mcvid/mcvid-timeout-functions.html) et au [suivi de la génération d’identifiants de visiteurs côté client](https://marketing.adobe.com/resources/help/en_US/mcvid/mcvid-client-side-id.html). (MCID-116)

**Problèmes connus**

Les clients qui utilisent le code DIL [!DNL  Audience Manager] et le code visitorAPI.js sur la même page doivent définir la variable DIL ` secureDataCollection= false`. Voir [secureDataCollection](https://marketing.adobe.com/resources/help/en_US/aam/dil-secure-data-collection.html).

**Version de maintenance : 18 août 2016**

**Correctifs et améliorations**

* Ajout de nouvelles fonctions de création de rapports aux [!UICONTROL audiences adressables]. (AAM-24411)
* Correction d’un bogue du créateur de segments qui empêchait d’ajouter des caractéristiques aux nouveaux segments dans Chrome. (AAM-24994)
* Correction d’un bogue en raison duquel les administrateurs ne pouvaient pas affecter d’autorisations aux groupes contenant un grand nombre de sources de données. (AAM-24870)
* Correction d’un bogue en raison duquel les dossiers de segments ne pouvaient pas être développés pour afficher les sous-dossiers ou segments. (AAM-25054)
* Correction d’un bogue du [!UICONTROL rapport d’état à l’intégration] qui décalait la plage de dates d’un jour avant la date sélectionnée. (AAM-25484)
* Correction d’un lien rompu vers la documentation d’aide dans les [!UICONTROL paramètres de source de données]. (AAM-25447)

**Correctifs de bogues DIL, améliorations et code obsolète**

Le code DIL constitue le code de collecte de données d’API de [!DNL  Audience Manager]. Dans la version de juillet, la dernière version du code DIL (version 6.5) est utilisée ; elle permet de profiter des fonctions et correctifs ci-après.

* Correction d’un bogue en raison duquel le code DIL ne pouvait pas être déployé dans Firefox. (AAM-24708)
* Ajout d’un attribut de titre à l’iFrame DIL. Avec un titre d’iFrame, les lecteurs d’écran peuvent plus facilement fournir des informations sur la page aux utilisateurs qui ont besoin d’aide lors de leur interaction avec du contenu en ligne. (AAM-25317)
* Prise en charge du code DIL pour le partage des ressources Cross-Origin (CORS). Voir [useCorsOnly](https://marketing.adobe.com/resources/help/en_US/aam/dil-use-cors-only.html). (AAM-25319, AAM-20139).
* L’iFrame DIL prend en charge les attributs ` iframe.sandbox='allow-scripts allow-same-origin';`. Le fait d’autoriser seulement ces deux jetons renforce la sécurité. Pour plus d’informations, voir la section sur les attributs de cette [documentation sur les iFrames](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/iframe). (AAM-23335)
* Le code DIL peut maintenant émettre des appels HTTPS sécurisés par défaut. Les clients qui utilisent le code DIL [!DNL  Audience Manager] et le code visitorapi.js sur la même page doivent définir la variable DIL ` secureDataCollection= false`. Voir [secureDataCollection](https://marketing.adobe.com/resources/help/en_US/aam/dil-secure-data-collection.html).

**Correctifs, améliorations et obsolescences**

* Augmentation de la fréquence et de la durée de vie de la collecte des données de mesure Facebook. Les nouvelles échéances de collecte des données sont les suivantes :

   * Après une publication, toutes les mesures Facebook sont collectées chaque heure pendant les 72 premières heures.
   * Ensuite, pendant les 72 heures suivantes, la collecte des données a lieu toutes les 8 heures.
   * Puis, pendant les 60 jours suivants, la collecte de données a lieu une fois par jour.

* Ajout d’une colonne de géociblage pour Twitter dans le rapport Analyses de publication exporté.
* Cinq nouvelles mesures de réaction Facebook sont désormais disponibles dans les rapports Analyses de publication et Analyses de propriété : J’adore, Haha, Wouah, Triste, Grrr. Dans une prochaine version, les mesures de réactions Facebook seront également disponibles dans les rapports Adobe Analytics.
* Correction d’un problème en raison duquel le fait de cliquer sur la loupe pour une publication Twitter privée dans [!DNL  Social Buzz] renvoyait en boucle à la même page plutôt que d’ouvrir la publication originale dans Twitter.
* Correction d’un problème en raison duquel il n’était pas possible de réclamer certains tweets réaffectés.
* Correction d’un problème en raison duquel certaines publications étaient perdues dans les rapports Buzz social générés différents jours.
* Correction d’un problème en raison duquel les paramètres de flux modifiés dans la modération unifiée n’étaient pas enregistrés après déconnexion puis reconnexion.
* Correction d’un problème en raison duquel les URL dans les tweets cités comprenaient plus de 24 caractères comme prévu.
